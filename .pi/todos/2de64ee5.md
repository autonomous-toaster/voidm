## Overview
Replace fixed 400-char truncation with intelligent passage extraction. Extract relevant passages containing query terms, then pass only those to reranker. This leverages cross-encoder strengths properly without information loss.

## Problem Statement
Current approach (400-char truncation):
- ✅ Works for documents where query appears early
- ❌ Loses information if query appears later in document
- ❌ Doesn't work well for multi-topic documents
- ⚠️ Arbitrary length limit

Ideal approach (passage extraction):
- ✅ Finds actual relevant passage containing query terms
- ✅ No information loss
- ✅ Works on any document length
- ✅ Respects document structure

## Implementation Plan

### Phase 1: Core Passage Extraction (Day 1)
**Goal**: Extract sentences/passages containing query term matches

1. **Implement sentence tokenization**
   - Split document into sentences
   - Preserve sentence boundaries
   - Handle abbreviations (Dr., Mr., U.S., etc.)
   - Location: New function `extract_passages()` in search.rs

2. **Find query-matching sentences**
   - Simple keyword matching: sentence.contains(query_term)
   - Case-insensitive matching
   - Handle query with multiple terms (match any term)
   - Example: query="machine learning" → find sentences with "machine" OR "learning"

3. **Extract passage with context**
   - Target passage: sentence(s) containing match
   - Add context: ±1 sentence before/after (configurable)
   - Result: 200-400 char passage (usually 2-4 sentences)
   - Config option: `context_sentences` (default: 1)

4. **Test coverage**
   - Test: Single-term query in early/middle/late sentence
   - Test: Multi-term query
   - Test: Multiple matches (pick best/closest)
   - Test: No match fallback (use first N chars like now)

### Phase 2: Integration with Reranker (Day 2)
**Goal**: Replace truncation with passage extraction in apply_reranker()

1. **Update apply_reranker() function**
   ```rust
   // Old: let docs_to_rerank = results[..apply_to_k]
   //      .map(|r| safe_truncate(&r.content, 400))
   
   // New: let docs_to_rerank = results[..apply_to_k]
   //      .map(|r| extract_best_passage(&r.content, &query, config))
   ```

2. **Add configuration options** in SearchConfig:
   ```toml
   [search.reranker.passage_extraction]
   enabled = true                    # Use passage extraction instead of truncation
   context_sentences = 1             # Sentences before/after match to include
   fallback_length = 400             # If no match found, use first N chars
   min_passage_length = 50           # Don't return passages shorter than this
   ```

3. **Update logging**
   - Log extracted passage for debugging
   - Log if fallback was used (no match found)
   - Help users understand what's being reranked

4. **Backwards compatibility**
   - If passage_extraction.enabled = false, use old 400-char truncation
   - Default: true (new behavior)
   - Users can opt-out if issues arise

### Phase 3: Advanced Features (Day 3, Optional)
**Goal**: Improve passage quality for better reranking

1. **BM25-aware passage extraction**
   - Use existing BM25 scoring to find best-scoring sentences
   - Prioritize sentences with query terms that had high BM25 scores
   - More sophisticated than just "contains term"

2. **Multi-match handling**
   - If query appears in multiple places, extract best match
   - "Best" = highest BM25 score, or closest to start, or most density

3. **Passage overlap prevention**
   - If extracting multiple passages, ensure they don't overlap
   - Or allow overlap but deduplicate sentences

4. **Caching passages** (performance optimization)
   - Cache extracted passages in memory
   - Key: (document_id, query_hash)
   - Avoid re-extracting same passage for repeated queries

### Phase 4: Testing & Validation (Ongoing)
**Goal**: Verify extraction works correctly and improves reranking

1. **Unit tests**
   - Test sentence splitting on various inputs
   - Test query matching (single/multi-term)
   - Test passage extraction with/without matches
   - Test edge cases (very short doc, very long doc, no matches)

2. **Integration tests**
   - Full search → rerank pipeline with extraction
   - Verify extracted passages are reasonable
   - Check scoring improvements vs truncation

3. **Benchmarks**
   - Performance: extraction time per document
   - Reranking quality: compare extracted passage vs truncated
   - End-to-end latency impact

4. **Manual testing**
   - Test on actual voidm memories
   - Compare old (truncation) vs new (extraction) results
   - Example: "python" search should find all relevant passages, not just first 400 chars
